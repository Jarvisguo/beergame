# 啤酒分销游戏 — 需求规格说明

> 本文档作为服务端重构的规格说明。UI（public/ 目录）保持不变，重构范围为 src/ 目录。

---

## 1. 系统概述

啤酒分销游戏（Beer Distribution Game）是一个多人在线供应链模拟游戏。每个团队由 4 名玩家组成，分别扮演供应链中的不同角色，通过 26 周的运营决策体验"牛鞭效应"。

---

## 2. 游戏规则

### 2.1 基本参数

| 参数 | 值 |
|------|----|
| 每队人数 | 4 人 |
| 游戏周数 | 26 周 |
| 初始库存 | 12 箱 |
| 初始吞吐量 | 4 箱/周 |
| 库存成本 | 0.5 元/箱/周 |
| 积压成本 | 1.0 元/箱/周 |
| 订单延迟 | 1 周（下游订单在下一周被上游收到） |
| 物流延迟 | 2 周（上游发货在 2 周后被下游收到） |

### 2.2 角色与供应链层级

```
客户需求 → 零售商 → 批发商 → 区域仓库 → 工厂
```

| 索引 | 角色 | 下游 | 上游 |
|------|------|------|------|
| 0 | 零售商 | 客户 | 批发商 |
| 1 | 批发商 | 零售商 | 区域仓库 |
| 2 | 区域仓库 | 批发商 | 工厂 |
| 3 | 工厂 | 区域仓库 | 工厂自身（无限产能） |

### 2.3 每周操作流程

每周所有 4 名玩家独立提交订单后，服务端统一推进，按以下顺序处理每个角色：

1. **接收货物**：从物流队列（shipping）取出本周到达的货物，加入库存
2. **接收订单**：零售商从客户需求获取；其他角色从订单队列（mailing）取出
3. **发货**：`实际发货 = min(库存 + 积压订单, 库存)`，发货进入 shipping 队列（2 周后到达）
4. **更新库存和积压**：`新库存 = 旧库存 - 实际发货`，`新积压 = max(0, 需发货量 - 库存)`
5. **下单**：玩家提交的订单数进入 mailing 队列（1 周后到达）
6. **计算成本**：`本周成本 = 库存 × 0.5 + 积压 × 1`，累加到总成本

### 2.4 客户需求趋势

| 趋势 | 描述 | 数值序列 |
|------|------|---------|
| mixed（默认） | 先增后减 | 4→6→8→10→8→6→4（循环） |
| growth | 持续增长 | 4→6→8→10→12→14→16（循环） |
| decline | 持续下降 | 16→14→12→10→8→6→4（循环） |

## 3. 游戏状态机

### 3.1 全局状态

```
waiting → started → ended
                ↓
            (reset)
              waiting
```

| 状态 | 描述 |
|------|------|
| waiting | 游戏未开始，不允许玩家登录 |
| started | 游戏进行中，允许玩家登录和操作 |
| ended | 游戏已结束，不允许新操作 |

### 3.2 团队状态

每个团队独立维护：

| 字段 | 类型 | 描述 |
|------|------|------|
| week | number | 当前周数（0 = 未开始，1-26 = 进行中，27 = 已结束） |
| waitingForOrders | string[] | 本周尚未提交订单的角色名列表，固定从 BEER_NAMES 初始化 |
| users | GameUser[4] | 4 个角色槽位，按索引固定（0=零售商...3=工厂） |
| cost | number | 团队总成本 |
| costHistory | number[] | 每周团队总成本历史 |
| shipping | number[][] | 物流队列，每个角色 2 个槽位 |
| mailing | number[][] | 订单队列，每个角色 1 个槽位 |

### 3.3 waitingForOrders 规则

- **初始化**：每周开始时固定为 `['零售商', '批发商', '区域仓库', '工厂']`，不依赖当前在线玩家数
- **移除**：某角色提交订单后，从列表中移除该角色名
- **推进条件**：`waitingForOrders.length === 0` 时触发本周结算，推进到下一周

## 4. 玩家连接管理

### 4.1 登录流程

1. 全局状态为 `started` 时，玩家才能登录
2. 提交用户名后，服务端分配团队和角色槽位
3. 分配优先级：
   - **优先**：现有团队中 `socketId` 为空的槽位（顶替断线玩家）
   - **其次**：未满 4 人的团队的空槽位（不限周数）
   - **最后**：创建新团队
4. 同一用户名只能占一个槽位，不允许单人控制多角色

### 4.2 重连

- 同用户名重新连接，任何时刻均允许
- 恢复原有团队、角色、游戏数据
- 清除断线计时器

### 4.3 断线处理

1. 玩家断线后，启动宽限期计时器（默认 3 分钟，可通过 `MOBILE_RECONNECT_GRACE_MS` 环境变量配置）
2. 宽限期内重连：恢复原状态
3. 宽限期结束（`finalizeDisconnect`）：
   - 清除 `g.users[index].socketId`
   - 从 `users` 字典中删除该用户名（释放槽位给新玩家）
   - 游戏数据（库存、成本、历史等）保留在 `g.users[index]`

### 4.4 顶替规则

新玩家顶替断线槽位时：
- 继承原玩家的所有游戏数据（库存、成本、积压、历史记录）
- 仅更新用户名和 socketId
- `waitingForOrders` 不做修改（已正确反映该角色本周是否已提交）

### 4.5 第 0 周行为

- 第 0 周时，玩家处于等待大厅，不可提交订单
- 当团队凑满 4 人时，自动推进到第 1 周（`initGroup`）
- 客户端在 week=0 时隐藏操作界面，显示等待大厅

## 5. Socket 事件接口

### 5.1 玩家端 → 服务端

| 事件 | 参数 | 描述 |
|------|------|------|
| `submit username` | userName: string | 登录/重连 |
| `submit order` | amount: number | 提交本周订单 |

### 5.2 服务端 → 玩家端

| 事件 | 数据 | 描述 |
|------|------|------|
| `game can login` | — | 游戏开始，允许登录 |
| `game started` | { week, waitingForOrders, users, demandTrend, demandProfile } | 团队进入第 1 周 |
| `next turn` | { week, update, waitingForOrders } | 推进到下一周 |
| `game ended` | { groups } | 游戏结束 |
| `game reset` | — | 游戏重置 |
| `group member joined` | { idx, update } | 有玩家加入本团队 |
| `group member left` | { idx, update } | 有玩家断线超时 |
| `update order wait` | waitingForOrders: string[] | 有人提交订单，更新等待列表 |
| `player rejoined` | { idx, update } | 有玩家重连 |

### 5.3 管理端 → 服务端

| 事件 | 参数 | 描述 |
|------|------|------|
| `submit password` | password: string | 管理员登录 |
| `start game` | { demandTrend? } | 开始游戏 |
| `end game` | — | 结束游戏 |
| `reset game` | — | 重置游戏 |
| `remove group` | groupIndex: string | 删除指定团队 |

### 5.4 服务端 → 管理端

| 事件 | 数据 | 描述 |
|------|------|------|
| `update table` | { numUsers, groups } | 全量更新团队表 |
| `update group` | { groupNum, groupData, numUsers } | 单个团队更新 |

---

## 6. 管理端功能

- 密码验证（默认 `admin`，可通过 `ADMIN_PASSWORD` 环境变量配置）
- 选择需求趋势（mixed / growth / decline）
- 开始 / 结束 / 重置游戏
- 查看所有团队实时状态（周数、成本、各角色玩家、待下单角色、牛鞭效应）
- 删除团队（仅游戏未开始时）
- 打开经营报告页面

---

## 7. 报告页面功能

- 从 sessionStorage 读取团队数据
- 全局概览：总人数、冠军团队、平均/最高成本、牛鞭效应统计
- 每个团队的详情卡片：总成本、库存、积压、各角色牛鞭分
- 牛鞭效应计算：`CV = stdDev / mean`，与下游 CV 对比得出放大比率
- 打印功能

## 8. 重构目标与约束

### 8.1 目标

| 目标 | 说明 |
|------|------|
| 代码结构清晰化 | 将 server.ts 拆分为独立模块：游戏逻辑、状态管理、Socket 事件处理 |
| 可测试性 | 游戏逻辑（advanceTurn、registerUser 等）与 Socket.IO 解耦，可独立单元测试 |
| Bug 修复 | 修复断线顶替、waitingForOrders 固定化等本次会话中发现的问题 |

### 8.2 约束

- `public/` 目录（HTML/CSS/JS）保持不变
- Socket 事件名称和数据结构保持不变（客户端兼容）
- 环境变量接口保持不变（`PORT`、`ADMIN_PASSWORD`、`MOBILE_RECONNECT_GRACE_MS`）
- `index.js` 入口保持不变

### 8.3 建议模块划分

```
src/
├── config.ts          # 常量和环境变量
├── types.ts           # 类型定义
├── game/
│   ├── roles.ts       # 角色定义和初始化
│   ├── demand.ts      # 客户需求趋势
│   ├── turn.ts        # advanceTurn 核心逻辑
│   └── group.ts       # 团队初始化和重置
├── state/
│   ├── groups.ts      # 团队状态管理
│   └── users.ts       # 用户注册和断线管理
├── socket/
│   ├── player.ts      # 玩家端事件处理
│   └── admin.ts       # 管理端事件处理
└── server.ts          # Express + Socket.IO 入口（精简）
```

### 8.4 测试策略

- 游戏逻辑模块（game/）可纯函数化，直接单元测试，无需启动服务器
- 集成测试通过 `index.js` 启动完整服务，使用 socket.io-client 测试 Socket 事件流
- 测试入口统一使用 `node index.js`（已有 `index.js` 作为 tsx 入口）
